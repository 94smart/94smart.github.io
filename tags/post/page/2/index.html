页面制作，不，是前端

  • 多少年过去了，做网站还是得靠页面制作，不管是CSS还是JS，现在还得搞HTML5，只是职位改叫前端了；
  • 用CSS+JS代码完成针对屏幕尺寸的页面适配叫响应式设计了，前端更成了很多产品的命根子；
  • 原来以为WAP网页WML、2.0暂时用不着JS了，结果Jquery Mobile之类的来了，说：移动互联网我们也能动起来；
  • 现在有了PhoneGap和钛，用HTML5+JS就能搞手机应用了；
  • 现在有Node.js之类的，可以用服务器端运行JS搞些东西了；
  • 很多时候设计师还是只能望代码兴叹，最后也得逼自己学这学那的；
  • Javascript语法成了更有发展的玩意，这不仅讽刺而且夸张，但如果你是技术或设计，不懂一点JS，还真不好意思说自己是做网站的。

注：本人所在公司正在招移动广告的前端互动设计，懂HTML5和CSS、熟练使用jquery(及mobile)制作动画效果，能为移动版页面做好页面适配就行，有意者可以发邮件到我@gmail的邮箱：94smart。

请问你是否经历过这样的产品流程，如何解决的？

来自知乎的问题：请问你是否经历过这样的产品流程，如何解决的？（有图有真相） 存个挡，我在知乎的回答如下： 我主要角色是PM和运营，经历的项目都会延期，立项和产品设计阶段算是比较顺利，BOSS和RD都没太多意见（因为没有多少人会详细看需求文档），开发期就很难讲，各种情况都会出现，最后必然都会延期。 YY下解决方法：

  1. 立项时设定明确的目标，尤其是BOSS的目标，且团队所有人必须都明白、认可；不认可的话不如直接退出。
  2. 产品设计阶段，PM需求分析要更实际，粒度更合适，保证符合目标，团队所有人都可参与需求迭代。
  3. 如果BOSS懂产品，PM就听、想并和RD沟通、执行，用结果证明BOSS的想法对错，不同的BOSS结果不同；如果BOSS强势表达意见，PM只能解释、举例证明，没得商量的就照做，结果不好的话最后就是换公司。
  4. 项目需求分析必须设定截止期限，超期需求只能下期加入，BOSS也不例外，做不到这点的不如不做。
  5. 项目整体PM都要团结RD，了解RD的想法、难处，如果RD懂产品、执行力高，一般合作很愉快，否则只能看是否能说服。
  6. 做到上面的，开发期RD可以踏踏实实的开发，PM等角色辅助支持就好。

ScoreLoop也被收购，还是被黑莓的母公司

真是邪乎了，ScoreLoop这个德国的社交游戏平台（与OpenFeint同类），居然被RIM（黑莓母公司）收购了，2011年6月8日上午，法新社说本周二RIM发布的消息。 关于游戏社交平台：

  • Apple的Game Center刚出来时没多少人看好，但第三方平台用行动证明了官方平台的实力，后边Google也（会）有自己的平台，第三方平台要怎么办？
  • 去年10月Plus+归DeNA，今年4月OpenFeint被GREE收购，以及这次的ScoreLoop被RIM收购，截至今年、6月，前几个起来的智能手机游戏社交平台都无一例外的是被收购。
  • 现在看，第三方的Social Gaming最后不免是被收购的结局，产品想象力也就这样了？倒要看看今年资本追逐的热点在未来会怎样，后边的游戏社交平台是否能承载几个上市公司出来。

关于RIM、黑莓、移动设备：

  • 其实Apple、Google、BlackBerry、Nokia、微软等智能手机相关厂商，都在拼命的为自己的移动设备在社交平台方面挣得更多的份额，游戏、娱乐自然也是不能放弃的战场。
  • 结合RIM今年上半年的各种收购和黑莓的各种调整，黑莓明摆着是要发力社交平台和服务。
  • PlayBook的发布多少有些不顺，RIM看来要押宝游戏这门了，可以预见的是ScoreLoop会尽快开展iOS、Android、WP现有游戏应用转黑莓的服务。
  • 一直觉得游戏、娱乐不是黑莓擅长的事情，也许PlayBook更适合，拭目以待吧。

p.s.年初因为当时工作的关系，关注了一下这类应用和服务，对用户来说现在还不是特别的重要，多数平台都是行业内部消化，像RIM这种收购很少见，最后还是祝他们幸福，没准可以给行业带来新的组合。
