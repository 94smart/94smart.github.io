比较郁闷的MM_preloadImages问题

最近工作的过程中发现用Javascript做图片预读取总是出问题：

  1. 做个展开菜单，展开项目的底图总是不能直接显示出来。
  2. 做个缩略图切换，左右切换的时候，新换出来的图片总是不显示，非得使用鼠标右键选显示图片才行
  3. ……

代码逻辑应该没问题，所以最后归结到使用的MM_preloadImages()上，不知道为什么好像没起作用，没能在页面显示之前预读取所需图片。

简述一下我的用法，需要预读的图片有10张，我使用的是Dreamweaver 8.0提供的v3.0版本的代码，代码如下：

function MM_preloadImages() { //v3.0 var d=document; if(d.images){ if(!d.MM_p) d.MM_p=new Array(); var i,j=d.MM_p.length,a=MM_preloadImages.arguments; for(i=0; i<a.length; i++) if (a[i].indexOf("#")!=0){ d.MM_p[j]=new Image; d.MM_p[j++].src=a[i];}} }

自动生成的预读调用是在body的onload上，我将它改为.js文件，在head里调用，又编写了一个预读函数用window.onload调用，代码如下：

function doPreload(){ for (var index=0; index<10; index++){ url1 = path_prefix+'TN_'+index+path_ext; MM_preloadImages(url1);} }

window.onload = doPreload;

运行时并没有得到预期效果，Firefox正常，IE出错，于是我改回body调用，还是失败。我以为是函数和循环的问题，就改成MM_preloadImages(‘img1.gif’, ‘img2.gif’, …)的调用，结果还是不行。

参考了一下Google Group里的文章，说有可能是缓存图片容量太大，可是回头想想一共10张图片，每张3、4K，全读出来也就30、40K，现在的带宽肯定不成问题。

所以我现在很郁闷，不知道其他人是不是也遇到这种问题，欢迎大家来与我交流，同时希望Javascript高手可以不吝赐教，指点我一下。

UPDATE：多谢Reg提醒，让我找到了问题的方向，看样子不是MM_preloadImages预读的问题，而是调用时候的问题，上面的懒得改了，就在下面继续更新。

**UPDATE2：**看到一篇讲CSS在IE里面的重复渲染问题，说DIV的显隐会影响CSS重新渲染，我不知道做的那个菜单是不是这种情况。

糟糕的体验：北京公交一卡能通？

由于种种原因，我最近两天才开始使用公交一卡通坐公共汽车，所以这个糟糕的体验来得晚了一点。

根据两天的感受，我觉得公交这次的动作很失败，费力不讨好，总的感觉是：公共汽车越来越堵（等着乘客刷卡）、越来越挤（排队刷卡？），售票员越来越忙（上车盯着卡，下车盯着票），乘客越来越麻烦（卡得掏出来塞回去的），浑水摸鱼的人多起来了。

下面捡几个点说，欢迎补充：

测试没做好，不管这个系统做的怎么样，它很缺练，从乘客和售票员的手忙脚乱中看出，没有做好内部测试，虽然以前看过刷卡机上贴着"测试中，请勿刷卡"什么的，它做的测试可能只是硬件的，并没有怎么用人去测。

举个不大好的例子，网络游戏都要经过内测、外测才收费，这个时下青少年朋友都知道的道理，公交愣不知道？一个没有经过压力测试的系统就敢让北京这么多人掏钱用吗？

规则不统一，现在存在多种规则，上车刷卡（单一票制）、上下车都刷卡（按里程制公交车、城铁）和下车刷卡（出租车），每一种对单一人群都是合理的，但是混合起来就晕死了。

为什么不能统一这个规则呢？比如全部实行上下车刷卡，这样不用去考虑到底下车的时候要不要刷卡，也不会忘记下车刷卡，计算全部由系统来做，最起码乘客解脱了。

缺乏异常处理，刷错了怎么办？刷多了怎么办？下车忘了刷卡怎么办？之后怎么补救？怎么减少自己的损失？这些真的都有吗？

北京公交一卡真的能通？

一年一次的中招：遗憾

又中招了，要自爆，上次是怪癖，这次是遗憾。写Blog后我从没有后悔，写写遗憾也好，同时可以让自己回忆一下过去，还不错。

这次是风在南方留言要我参加，我就接着自爆一下吧。不过这个游戏传的够快的，昨天才在ittalk那里看见，谁知今天就找到我了。

先说一下规则：

游戏规则－遗憾接龙－“在写下本人的几个遗憾之后，将这个游戏传给另外五个Blog。遗憾可以包括当初可以做但没有做以至于今天都不能再做的事情，也可以包括当初可以做但没有做今天要努力做的事情。并且，请书写这个遗憾的理由。至于遗憾的数量，一到五个随意，毕竟，不是人人都有那么多的遗憾，特别是对于年纪很轻的人来说。

特别声明一下：有些人会声称此生无憾，或者说有什么好遗憾的，现在去做也行啊。又或者是天天忙得要命，哪里有空遗憾。如果不愿意写遗憾的话，务请写上：本人没有遗憾，并把这个游戏传下去。多谢！”

以下是我的遗憾：

  1. 以前做了几个网站，有音乐下载的、俱乐部的、资讯的等等，都因为自己的原因没坚持下去。
    遗憾原因：很简单，要是当时接着做，没准现在已经拿到VC了，^_^

  3. 之前呆过的一家公司的期权，到走的时候都没有卖出去，期限OK，但是价格不OK。
    遗憾原因：头一次有了期权，但是没转换成$

Zoundry新版进化

几天没在，Zoundry就出了个新版，这版跨过了1.0，直接出了个1.0.23，修正了一批Bug，还推出了一些新功能，看来Zoundry还想将Beta继续下去，我原以为过了1.0会把Beta撤下来~

看看都有什么改进：

  1. 拼写检查（词典要自己下载）
  2. XHTML源码编辑及自动完成
  3. XHTML代码规范验证工具
  4. XHTML整理操作
  5. 支持语言本地化
  6. 查找/替换
  7. 提供MovableType API和WordPress文本实体扩展（例如摘要操作）
  8. 增加对Blog实体的排序方法（如按时间、按字母顺序等）
  9. 建立Blog帐号的时候增加支持RSD来识别API设置
  10. 其它修改，包括图片快速对齐菜单，用"Link to Posts"增加旧帖子的链接等

原文：http://www.zoundry.com/releasenotes.html

这个新版改动还是挺大的，主要是增加了源码编辑操作，又向Dreamweaver迈进了一步~界面改动不太大，用起来还是挺习惯的，希望下次改进不要再改动编辑区了，越改越复杂。

最令我感觉人性化的操作是崩溃恢复了，刚才写这篇的时候使用了XHTML代码验证，还使用的是严格式（Strict）的，一点就报错还自己关闭了，然后我就傻眼了，写的内容都没保存，结果再次打开Zoundry，发现内容还好好的在编辑区里，真是高兴死了。

还有就是本地化方面，Zoundry并没有采取其它软件的本地化策略，没有提供语言包，而是提供了开放型的解决方法，让用户去自己解决，想改什么就改什么。不过我觉得这种方法有点取巧，虽然很快就会有用户自己定制的语言包出来，但是却让大部分用户决得不够方便。

总的来讲，Zoundry在进化，但是方向并不是一直朝Blog编辑器去的，而是向专业网页编辑器（Dreamweaver），或者更夸张点说是向专业文本编辑器（Word？）去的，越做越高端，不清楚是不是要走高端路线了。

不过话说回来，这个编辑器还是值得继续推荐的，快下手~
