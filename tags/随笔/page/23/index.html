关于网页设计和CSS+XHTML

最近老冒写了两篇（1，2）关于Web2.0网站设计的文章，很多内容都有同感，尤其是第二篇里的：

**试图用css编码， 而不是用css设计
**设计人员仍然是photoshop画页面，然后切割，然后用css去实现。 这个往往导致的结果就是产生大量不必要的垃圾代码，xhtml的代码结构为了实现美工设计的某些其实微不足道的效果而变得复杂不堪。

感觉上面写的就和自己的工作及其相似：制作人员一遍又一遍得用CSS去实现Photoshop的设计，垃圾代码是不可避免的。

反过来想想，为什么不能直接让设计人员用CSS去设计网站呢？

以上观点有人持反对意见：

对美工而言，精通CSS，然后考虑到CSS的各种属性和限制去做设计，可能会限制他的创造力。所以我以为美工更应该清晰页面的结构，在页面设计的过程中，做到针对结构设计更为实际。在维持一个大局风格基础上，在类似的结构上有意识的样式统一，同时页面各个部分的承启转合清晰一致，那么页面制作人员也能方便的进行页面拆合，维护起来也方便。
来源：“web 2.0时代的网页设计风格” 读后感

我的观点是：使用CSS设计网站确实会限制创造力，这没错，但是功能服务性网站比较特殊，所以发挥创造力需要限制在一定范围里，这个范围包括对用户的易用性、可用性，以及对网站的结构化和可维护性。

换句话，就是在CSS正常发挥布局作用的前提下，在一个又一个的框框（也就是DIV）里面，发挥设计人员的最大创造力（美观和易用），这样可以尽量减少垃圾代码的堆积。

另外，老冒的想法有点过于理想了，“让更多的程序员参与到设计之中,让整个项目的人都think in css”，但是我真的希望所有相关人员都可以Think in CSS。

p.s.对于技术和美术的页面问题，包括CSS和XHTML，我现在的解决办法是，美术用CSS和XHTML制作页面，然后将页面根据DIV布局切块，每块里面只有纯XHTML的代码（可以适量有class和id），然后由技术人员针对单块内容的规则进行程序嵌入，最后由制作人员进行整合。这样做很麻烦，但是可以减少部分不必要的麻烦。

FEED43，没有Feed创造Feed也要订

现在大部分空闲时间都在读Blog，这正好也是自己写得少的一个借口（或者说原因），另一个借口是懒得写，但是今天（应该是前几天）想把最近的少少经验写一写。

首先要说明的是，我花费时间阅读的顺序由高到低是网站浏览、阅读器的Feed、Gmail邮件订阅，产生网摘（或收藏）的顺序是阅读器的Feed、网站浏览、Gmail邮件订阅。

也就是说，我花费了很多时间在网站浏览，收效还不如Feed阅读，我总结这种情况的出现主要是因为不得不去看一些不支持Feed的网站，但是这些内容的过滤并不比过滤Feed和Gmail容易，因为多数只有一个标题，没有摘要可以预览，所以都是一篇篇的打开看，这样的效率确实挺低的。

所以我急需一个将网站页面内容抓取然后Feed化的工具，正好FEED43就是。

扯了这么久终于可以进入正题了，先看看FEED43首页宣传用的图：

看了图，基本上不用解释，大家就知道FEED43是用来干什么的了吧，我就是看到这张图才决定要用的。不过我还得友情来个提醒，不要被这张图骗了（？！），因为实际并不是如图那样可视化操作的，而是用输入框操作的，且对HTML代码还得有一定了解。

简述一下过程：

  1. 先输入目标页面链接及其编码，会返回该页面的HTML源码
  2. 划定范围，找到内容区域的起始及结束代码，并插入通配符{%}，比如{%}
    确定重复出现内容的格式，并为将各内容文字替换为通配符{%}，比如{%}{%}
    FEED43会根据设定去分析HTML，完成后会得的一系列item，每个item会有对应的{%N}字符串，比如{%1}=http://xxx.com，{%2}=title，{%3}=text
  3. 接着设定Feed的标题、网页链接、描述
    然后设定Feed项目模板，包括标题、链接、内容/摘要，与之前的{%N}相对应。
    这里还有一个选项可以将所有item合并到同一个item里面，结果有点像我用365key生成的昨日收集
  4. 你将得到一个数字文件名的Feed地址，接着可以修改文件名、设定Feed修改密码、设定Feed为你私有等。

嘿，我用这个工具搞定了FTchinese。

正所谓，有Feed要订，没有Feed创造Feed也要订~

如果你也有跟我一样的烦恼可以去试试这个FEED43。

p.s.如果有Bookmarklet之类的工具就更好了，看到什么就截下来然后做成Feed。

Prototype1.4.0说明文档showLocalLinks例子有问题

链接在这里：http://www.sergiopereira.com/articles/prototype.js.html#EnumeratingArrays

这个showLocalLinks例子有问题，在哪个浏览器里都显示不出结果。

<script> function showLocalLinks(paragraph){ paragraph = $(paragraph); var links = $A(paragraph.getElementsByTagName('a')); //find links that do not start with 'http' var localLinks = links.findAll( function(link){ var start = link.href.substring(0,4); return start !='http'; }); //now the link texts var texts = localLinks.pluck('innerHTML'); //get them in a single string var result = texts.inspect(); alert(result); }

这个例子的意思应该是取出段落里面的本地链接，原理是链接开头不是http的就算，但是段落里面使用的#链接形式，无论在IE还是firefox里，都会被解释为带有http的形式，所以这个例子的结果永远都是空。

解决方法是把其中#开头的链接改为"javascript:;"，就可以看到效果了。

后话，也有可能是因为我用Apache调试的原因，#xxx全被解释为"http://localhost/#xxx"了。

toread，用Email做文摘

toread，一个使用Email做网摘的网站，使用它提供的Bookmarklet可以瞬间把页面内容发送到你的邮箱，然后你可以用收信软件把内容下载下来保存。

总体感觉干净，快捷。缺点是没有了Del.icio.us的分享特性。

起先在lifehacker.com上面看到了，没提起太大兴趣，后来在iThink的Blog又看到了，这次介绍的角度不同，因为iThink提到了Gmail，嘿，想到能把它和Gmail的Label过滤配合起来用真是太棒了，赶紧在Gmail里加了一个2read的Label，以后就可以直接将一些网页资料收藏到Gmail Space里面保存了，cool~

操作真的很简单，填邮箱注册，点确认信里的链接，将Bookmarklet加入收藏夹，以后想摘页面就可以直接用Bookmarklet操作。插一句，我使用FireFox的Bookmarklet越来越多，也越顺手，感觉比用IE下面的舒服太多了。

最后，发现其实不是toread多么的有创意，而是Google让这变得可能，现在也更加的佩服Google。

p.s.Google昨天发布的中文名感觉并不理想，但是也可以接受。

你也来一只Web袋鼠吧

Webaroo，一个离线搜索/浏览工具，我管它叫网页袋鼠。它所提供的功能重点是在离线的情况下也可以搜索，可以像袋鼠的育儿袋一样带着内容到处走，就如它所倡导的"Search unplugged"（搜索不插电/线？）。

Webaroo支持两种形式的离线搜索，分别是"Web Sites（网站）“和"Web Packs（网站包）"。网站方式跟其他离线浏览工具的操作类似，输入网址，选择目录层次等就开始索引了；网站包方式就是使用由Webaroo准备好的内容包，比如10大城市的相关内容、2006世界杯资讯、世界新闻等等，现在内容源还是不够多，估计以后会增加。

安装Webaroo之后，会增加一个类似Google桌面搜索的索引服务，不同的地方是Google索引本机的内容，Webaroo索引、保存目标网站的内容。索引的速度有点慢，感觉据他们宣称的"Highest Quality（高品质）“还是有段距离的。

Webaroo总体感觉还算不错，不过缺点也是存在的，它保存的页面好像都不支持CSS，所以内容看起来有点不顺眼。

现在Webaroo已经可以在手持设备上用了，这点倒不错，如果用的是Windows系统的手机/PDA，可以直接把保存的内容传进去，这样不用电脑也可以查看最新消息了，这么操作应该比直接用手机上网划算吧。

感兴趣的赶紧去领一只吧，现在还免费~
